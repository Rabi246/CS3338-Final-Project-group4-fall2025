\section{Introduction}

\subsection{Purpose of the Document}
The purpose of this Software Requirements Specification (SRS) is to define the functional 
and non-functional requirements for the Intelligent Document Processing \& Case File 
Search System. The software enables users to upload PDF case files, extract simulated 
metadata, store it in a PostgreSQL database, and search stored records. This document 
serves as a guideline for developers, instructors, and future maintainers.

\subsection{Intended Audience}
This SRS is intended for:
\begin{itemize}
    \item Developers implementing frontend, backend, and database components.
    \item Instructors evaluating the project's correctness and completeness.
    \item Students reviewing expected system behavior and features.
\end{itemize}

\subsection{Overview of the Software}
The system provides:
\begin{itemize}
    \item A PDF upload interface.
    \item Automatic extraction of basic (simulated) metadata.
    \item Structured storage in a PostgreSQL database.
    \item A search interface to retrieve case information.
\end{itemize}

The entire system runs inside Docker containers for consistency and ease of deployment.

\newpage

\section{External Interface Requirements}

\subsection{User Interface}
The software includes the following UI components:
\begin{itemize}
    \item \textbf{Upload Page:} Allows users to select and upload PDF files.
    \item \textbf{Search Page:} Enables case searching by ID, name, or keywords.
    \item \textbf{Case Detail Page:} Displays extracted metadata for a selected case.
    \item \textbf{Navigation Bar:} Provides access to Home, Upload, and Search.
\end{itemize}

\subsection{Software Interfaces}
\begin{itemize}
    \item \textbf{Frontend → Backend:} Communicates via REST API using JSON.
    \item \textbf{Backend → Database:} Interacts with PostgreSQL using environment variables.
    \item \textbf{Backend → Filesystem:} Temporarily stores uploaded files during processing.
    \item \textbf{Docker Network:} All services communicate through Docker's internal network.
\end{itemize}

\subsection{Hardware Interfaces}
No hardware interfaces are required. All components operate inside virtualized Docker containers.

\subsection{Communication Interfaces}
\begin{itemize}
    \item HTTP-based communication through ports exposed by Docker.
    \item Localhost access for frontend and backend services.
\end{itemize}

\section{System Requirements}


\subsection{Functional Requirements}

\subsubsection*{FR1: Document Upload}
\begin{itemize}
    \item Users shall upload PDF files through the Upload page.
    \item The system shall validate file type before processing.
\end{itemize}

\subsubsection*{FR2: Metadata Extraction}
\begin{itemize}
    \item The system shall extract simulated metadata fields (case ID, parties, dates).
    \item The system shall return clear error messages if extraction fails.
\end{itemize}

\subsubsection*{FR3: Database Storage}
\begin{itemize}
    \item Extracted metadata shall be stored in PostgreSQL with unique identifiers.
    \item Duplicate file uploads shall be flagged or prevented.
\end{itemize}

\subsubsection*{FR4: Search Functionality}
\begin{itemize}
    \item Users shall search records by ID, name, or keyword.
    \item Matching results shall be displayed in a tabular format.
\end{itemize}

\subsubsection*{FR5: Case Detail View}
\begin{itemize}
    \item The system shall display complete metadata for any selected record.
\end{itemize}

\subsection{Non-Functional Requirements}

\subsubsection*{NFR1: Performance}
\begin{itemize}
    \item Metadata extraction shall complete within 5 seconds per file.
    \item Search queries shall return results within 2 seconds.
\end{itemize}

\subsubsection*{NFR2: Usability}
\begin{itemize}
    \item The system shall be easy to use for non-technical users.
    \item Navigation shall be simple and intuitive.
\end{itemize}

\subsubsection*{NFR3: Reliability}
\begin{itemize}
    \item The system shall recover automatically if any Docker container restarts.
    \item All containers must start reliably with \texttt{docker-compose up}.
\end{itemize}

\subsubsection*{NFR4: Security}
\begin{itemize}
    \item Uploaded documents must not be accessible outside the system.
    \item No data shall be sent to third-party APIs.
\end{itemize}

\subsubsection*{NFR5: Compatibility}
\begin{itemize}
    \item The system shall run on Windows, macOS, and Linux using Docker Desktop.
    \item The frontend shall support Chrome, Safari, and Microsoft Edge.
\end{itemize}

\section{Legal and Ethical Considerations}

\subsection{Data Storage and Privacy}
\begin{itemize}
    \item Metadata may include sensitive information requiring careful handling.
    \item All data is stored locally within Docker containers.
    \item Documents are never sent to external services.
\end{itemize}

\subsection{Ethical Use of Document Processing}
\begin{itemize}
    \item Metadata extraction must not misrepresent document contents.
    \item Users shall be informed when extraction is incomplete or inaccurate.
\end{itemize}

\subsection{Compliance Considerations}
\begin{itemize}
    \item Only simulated test data is used—no real case files.
    \item The system shall follow relevant academic data policies.
\end{itemize}
